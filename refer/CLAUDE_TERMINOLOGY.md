# Claude 용어 정리 — 층별 이해

> **목적**: Agent, Sub-agent, Hook, Skill, Harness 등 Claude 관련 용어가 서로 어떻게 다르고 어떻게 엮이는지 명확히 이해하기 위한 레퍼런스.
> **전제**: 용어들이 겹쳐 보이는 건 자연스러운 일. 같은 층에 있는 개념이 아니라 **서로 다른 층의 부품**인데, 문서들이 평행선상에 나열해버려서 그렇다. 층을 먼저 잡고 본다.

---

## 0. 큰 그림 — 4개의 층

```
  ┌─────────────────────────────────────┐
  │  [층 4] 사용자 입력 · 트리거           │  슬래시 명령, 자연어 요청
  ├─────────────────────────────────────┤
  │  [층 3] 하네스 (Harness)              │  CLAUDE.md, Skill, Agent, Hook, Command, MCP
  ├─────────────────────────────────────┤
  │  [층 2] 실행 환경 (Runtime)            │  세션, 컨텍스트 윈도우, 도구, 권한, 샌드박스
  ├─────────────────────────────────────┤
  │  [층 1] 모델 (Model)                   │  Claude Sonnet / Opus / Haiku
  └─────────────────────────────────────┘
```

**핵심 비유**: 모델은 "뇌"다. 하네스는 "몸·사무실·비서·규칙"이다. 같은 뇌라도 환경이 좋으면 훨씬 잘 일한다.

---

## 1. Model (모델)

실제 언어 모델 자체. `claude-sonnet-4-6`, `claude-opus-4-6` 같은 이름이 붙는 것. **이 층에서는 아무것도 건드릴 수 없다**. 우리가 하는 모든 설정은 모델 위·바깥의 층에서 이루어진다.

---

## 2. Session / Context Window

**세션**은 한 번의 대화 흐름. **컨텍스트 윈도우**는 그 세션에서 모델이 기억할 수 있는 토큰 한도(예: 1M 토큰). 세션이 끝나면 컨텍스트는 사라진다.

이 한계가 **모든 하네스 설계의 배경**이다. 컨텍스트가 무한하면 하네스 설계 자체가 필요 없다.

---

## 3. Tool (도구)

모델이 "바깥 세계에 영향을 주기 위해" 쓸 수 있는 함수들.

- Read, Write, Edit — 파일 조작
- Bash — 셸 명령 실행
- Grep, Glob — 검색
- WebFetch, WebSearch — 웹 접근

**비유**: 손. 모델은 텍스트만 내뱉을 수 있지만, 도구를 호출하면 실제로 파일을 읽고 명령을 실행한다.

모든 도구 호출은 **권한(permissions)**을 통과해야 실제 실행된다. 차단되면 호출해도 실행 안 됨.

---

## 4. Harness (하네스)

**모델을 제외한 모든 것의 총칭**이다. 특정 파일 하나가 아니라, 다음의 총합:

- CLAUDE.md (기억)
- Skill (지식 수납장)
- Agent (전문 비서)
- Command (사용자 매크로)
- Hook (자동 검문소)
- Permissions (도구 사용 허가 규칙)
- MCP (외부 도구 연결)

"Claude를 효율적으로 쓴다"는 말은 결국 **하네스를 잘 설계한다**는 뜻이다.

---

## 5. CLAUDE.md

세션 시작 시 자동으로 모델에게 읽히는 **상시 메모리**다. 한번 쓰면 그 프로젝트·사용자의 모든 세션에 적용된다.

**비유**: 사무실 벽에 붙여놓은 규칙판. 모델이 매 대화마다 보고 시작한다.

**위치**:
- `~/.claude/CLAUDE.md` — 모든 세션에 적용 (개인 규칙)
- `프로젝트/CLAUDE.md` — 해당 프로젝트에만 적용 (팀 규칙)

**주의**: 길면 오히려 나쁘다. 토큰을 매 세션마다 잡아먹는다. 50~150줄 이내 권장.

---

## 6. Skill

**트리거될 때만 펼쳐지는 지식 패키지**다. 평상시에는 이름과 설명(description)만 노출되고, 관련 작업이 시작될 때 본문이 로드된다.

**비유**: 사무실 책장에 꽂힌 매뉴얼. 평소엔 제목만 보이고, 필요할 때 뽑아서 읽는다. "PPT 만드는 법", "PDF 처리 절차" 같은 대형 지식이 이상적.

**구조**:
```
skills/
└── my-skill/
    ├── skill.md        ← 본문 (트리거 시 로드)
    └── evals/          ← 선택: 테스트 케이스
```

트리거는 `skill.md`의 frontmatter에 있는 `description`을 보고 모델이 판단한다. 그래서 **description 품질이 전부**다.

---

## 7. Agent / Sub-Agent

**Agent**는 독립된 컨텍스트에서 돌아가는 "제2의 Claude 세션"이다. 메인 세션이 Task 도구로 호출한다.

**비유**: 옆방 비서에게 "이 파일들 보고 요약해 와" 시키는 것. 비서는 자기 방에서 100개 파일 읽고 돌아와서 요약 3줄만 보고한다. **메인 사무실은 어지럽혀지지 않는다** — 이게 agent의 핵심 가치.

**Sub-agent**는 "agent가 호출한 또 다른 agent"다. 용어가 위계를 나타내는 거지 기술적으로 다른 종류가 아니다.

vmsworks 예시:
```
dev (메인 skill)
  └─ feature-dev (agent)
        ├─ be-agent (sub-agent)
        └─ fe-agent (sub-agent)
```

**Agent 파일**:
```
agents/
└── code-reviewer.md    ← frontmatter + 본문
```

**Skill과의 차이**:
- **Skill**은 "지식"이다. 메인 모델이 그 내용을 읽고 **스스로** 행동한다.
- **Agent**는 "실행자"다. 별개 세션에서 **독립적으로** 일하고 결과만 돌려준다.

---

## 8. Command (슬래시 명령)

사용자가 `/` 뒤에 입력해서 **명시적으로 호출**하는 매크로다. `/commit`, `/review` 같은 것.

**비유**: 키보드 단축키. "이 순서로 일해달라"는 정형화된 요청을 짧은 이름 하나로 부른다.

**Skill/Agent와의 관계**:
- **Skill**: 모델이 자동 감지해서 끌어옴
- **Command**: 사용자가 직접 호출
- **Agent**: 메인 세션(또는 Command, 또는 Skill)이 Task 도구로 위임

같은 기능을 Command로도 Skill로도 만들 수 있다. **호출 방식 차이**.

---

## 9. Hook

**특정 이벤트 직전·직후에 자동 실행되는 스크립트**다. 모델이 끼어들 수 없는 결정론적 검문소.

**이벤트 예시**:
- `PreToolUse`: 도구 호출 직전 (차단 가능)
- `PostToolUse`: 도구 호출 직후 (포맷터 실행 등)
- `UserPromptSubmit`: 사용자가 메시지 보낼 때
- `Stop`: 세션 종료 시

**비유**: 공항 검색대. 모델이 "이거 지나갑시다" 해도 훅이 "안 돼요" 하면 안 된다.

**"프롬프트로 금지"와 다른 점**: 진짜로 막힌다. 프롬프트는 확률적 준수, 훅은 결정론적 차단.

---

## 10. MCP (Model Context Protocol)

**외부 서비스를 도구로 연결하는 표준 프로토콜**이다. Slack, GitHub, Notion, 데이터베이스 등을 MCP 서버로 감싸면 Claude가 내장 도구처럼 호출할 수 있다.

**비유**: USB-C 표준. 서비스 공급자가 MCP 서버를 만들면, 어느 Claude든 꽂아 쓸 수 있다.

---

## 11. Plugin

**Skill·MCP·명령어를 한 묶음으로 배포하는 단위**다. "React 개발용 플러그인"을 설치하면 관련 스킬·MCP가 한 번에 들어온다. 아직 성숙한 에코시스템은 아니지만 점점 중요해지고 있다.

---

## 12. Output Style

**응답 톤·형식의 프리셋**이다. 상황별로 간결 모드·심층 모드를 전환할 때 쓴다.

**CLAUDE.md vs Output Style**:
- CLAUDE.md: 항상 적용
- Output Style: 지금만 적용

---

## 13. Permissions (권한)

각 도구에 대해 **allow / deny / ask** 3단계로 접근 제어하는 규칙. `settings.json`에 선언한다. 훅과 함께 안전의 두 축이다.

- **allow**: 묻지 않고 실행
- **deny**: 무조건 차단
- **ask**: 매번 사용자에게 물어봄

---

## 14. Sandbox

명령 실행이 **격리된 환경**에서 돌아가는 것. 호스트 시스템을 오염시키지 않는다. 컨테이너 안에서 어떤 위험한 명령을 실행해도 바깥으로 새지 않는다.

---

## 관계도 — 한 장면으로 보기

사용자가 "API 엔드포인트 추가해줘"라고 입력했다고 치자:

```
사용자 입력
   │
   ▼
[UserPromptSubmit Hook]  ← "배포" 같은 키워드 검사 (여기선 통과)
   │
   ▼
메인 세션 시작
   ├─ CLAUDE.md 로드 → 프로젝트 규칙·금지사항 상주
   ├─ 관련 Skill 자동 매칭 → "api-design" skill 본문 펼침
   │
   ├─ 탐색 필요 → Task 도구로 Sub-agent 호출
   │      └─ explainer agent가 관련 파일 10개 탐색 → 요약 반환
   │
   ├─ 편집 시작 → Edit 도구 호출
   │      ├─ [PreToolUse Hook] 권한 확인
   │      ├─ 실제 편집 실행
   │      └─ [PostToolUse Hook] 포맷터 자동 실행
   │
   ├─ 리뷰 필요 → code-reviewer agent 호출 → 리포트 반환
   │
   └─ 커밋 → /commit command 호출
          ├─ [PreToolUse Hook] `git commit` 권한 확인
          └─ 실행
   │
   ▼
[Stop Hook] 미완료 TODO 체크
```

위 장면의 **모든 부품이 하네스**다. 모델은 이 오케스트라의 바이올린 한 대일 뿐이고, 나머지 악기·지휘자·악보가 **당신이 설계하는 부분**이다.

---

## 가장 흔한 혼동 3가지

### Skill vs Agent

**지식 vs 실행자**.

- Skill 본문은 **메인 세션**이 읽고 따른다. 메인 모델의 지식이 확장됨.
- Agent는 **별개 세션**에서 스스로 일한다. 메인 세션에는 요약만 돌아옴.

판단: 지식을 확장하고 싶으면 Skill, 작업을 격리하고 싶으면 Agent.

### Command vs Skill

**호출 방식 차이**.

- Command: 사용자가 직접 타이핑 (`/commit`)
- Skill: 모델이 description 보고 자동 감지

판단: 사용자가 명시적으로 부를 거면 Command, 상황 기반 자동 발동이면 Skill.

### Hook vs Permissions

**둘 다 게이트지만**:

- Permissions: 선언적 규칙 (allow/deny 리스트) — 단순하고 빠름
- Hook: 스크립트 (프로그래밍 가능한 로직) — 복잡한 조건 처리 가능

판단: 복잡한 조건은 Hook, 단순한 것은 Permissions.

## 한 문장 정리

| 용어 | 한 문장 정의 |
|------|-----------|
| Model | Claude 뇌 |
| Session | 한 번의 대화 흐름 |
| Context Window | 세션의 기억 용량 |
| Tool | 모델의 손 |
| Harness | 모델 바깥의 모든 것 |
| CLAUDE.md | 상시 메모리 (벽에 붙은 규칙판) |
| Skill | 트리거 시 펼쳐지는 지식 패키지 |
| Agent | 독립 세션에서 일하는 비서 |
| Sub-agent | Agent가 호출한 Agent |
| Command | 사용자가 타이핑하는 매크로 |
| Hook | 결정론적 자동 검문소 |
| MCP | 외부 도구 연결 표준 |
| Plugin | Skill·MCP 묶음 배포 단위 |
| Output Style | 응답 톤 프리셋 |
| Permissions | 도구 사용 허가 규칙 |
| Sandbox | 격리된 실행 환경 |

---

## 층으로 다시 정리

**Model · Session · Tool**은 기반 인프라.
**CLAUDE.md · Skill · Agent · Command · Hook · MCP**는 그 위에 쌓는 하네스.
**Permissions · Output Style**은 양쪽을 연결하는 조절기.

이 관점이 잡히면 문서를 읽을 때 각 용어가 어느 층을 말하는지 즉시 위치 잡힌다.
