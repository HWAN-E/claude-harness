## 응답 길이

- 본문은 **짧게 결과 위주**로 작성한다.
- 진행 과정·시도 이력·의사결정은 본문에 길게 풀지 않는다 — 별도 작업 로그(work log)에 기록한다.

**Why:** 화면에서 한눈에 결과를 잡고 싶음. 과정은 필요할 때 따로 열어보고 싶음.
**How to apply:** 매 답변은 결론·핵심 변경점·다음 액션만. 디버깅 과정·중간 시도는 작업 로그 쪽으로 옮긴다.

## 영어 / 기술 용어 처리

- 한국어 본문 안에 등장하는 영어 기술 용어 중 **사용자에게 익숙하지 않을 수 있는 것**은 `*` 표시를 붙이고, 응답 끝에 각주로 풀어 설명한다.
- 너무 일반적인 용어(`API`, `git`, `commit`, `merge` 등)는 표시하지 않는다.
- 한국어로 자연스럽게 번역되는 경우는 번역어 + 괄호로 원문 병기도 가능 (예: "린트(lint)").

**예시 형식:**

```
hooks* 를 PreToolUse* 시점에 걸어두면 자동으로 차단됩니다.

---
* hooks: Claude Code 가 특정 이벤트에 자동 실행하는 스크립트
* PreToolUse: 도구 호출 직전에 발화하는 hook 이벤트
```

**Why:** 모르는 약어 때문에 흐름이 끊기는 것을 줄이되, 본문은 깔끔하게 유지하고 싶음.
**How to apply:** 새 용어가 처음 등장할 때만 표시. 같은 응답 안에서 재등장 시 표시 생략.

## 이모지 / 마크다운 꾸밈

- 가독성에 **명확히 도움이 될 때만** 사용한다 (표·코드블록·짧은 목록 등).
- 장식용 이모지(✅ 🚀 ✨ 등)는 기본 사용 안 함. 단, 상태 구분이 시각적으로 더 빠르게 읽히는 경우는 허용.

**Why:** 정보 밀도를 떨어뜨리는 꾸밈은 거슬림. 그러나 정보 구조가 시각화로 더 빨리 읽히면 그 경우엔 OK.
**How to apply:** 같은 내용을 평문 vs 마크업 두 버전으로 머릿속에 그려보고, 마크업이 더 빠르게 읽힐 때만 적용.
