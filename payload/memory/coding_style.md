## 주석

- **의도/이유(Why)만** 적는다.
- "무엇을(What) 하는 코드인지" 는 식별자와 코드가 이미 드러내므로 적지 않는다.
- 비자명한 제약, 외부 시스템과의 약속, 워크어라운드 설명 같은 **컨텍스트가 비명시적**일 때만 작성.

## 테스트

- 변경 규모에 따라 판단:
  - 한 줄 수정·오타·UI 텍스트 변경 → **생략**
  - 새 함수·메서드 추가, 복잡한 로직, 버그 수정 → **단위 테스트 자동 생성**
  - 시스템 경계 변경 (API, DB, 외부 호출) → **통합 테스트 추가 검토**
- "테스트 짜지 마" 라고 명시하면 따른다.

## 커밋 메시지

- **Conventional Commits** 형식 유지:
  - 타입: `feat`, `fix`, `docs`, `refactor`, `chore`, `style`, `test`, `build`, `ci`, `perf`, `revert`
  - 영역(scope) 있으면 명시: `feat(web):`, `fix(api):`
- 제목은 한국어로, 50자 이내.
- 본문은 "왜" 가 비자명할 때만 작성.
- 사용자 사전 확인 없이 자동으로 commit 하지 않는다 (work_style.md 의 사전 확인 규칙 적용).

**Why:** 코드의 의도를 영구적으로 남기는 것은 주석·커밋 메시지·테스트뿐. 이 셋의 일관성이 미래의 가독성을 좌우한다.
